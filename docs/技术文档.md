# Kalm 技术文档

> 版本 1.0.0 | 2026-05-15

---

## 一、项目定位

**Kalm** 是一个纯粹的 AI 任务中转控制站。它位于前端与 AI 后端服务之间，只做三件事：

1. **接收** — 前端提交任务请求
2. **调度** — FIFO 排队 + 管理后端子进程生命周期
3. **透传** — 把后端返回的 JSON 原样传给前端，把后端文件字节流原样传给前端

### 不是什么

- 不是业务逻辑层（不处理音频、不改写 URL、不判断下载/代理模式）
- 不是文件存储（文件代理只是网络拓扑要求，不做缓存或持久化）
- 不是 AI 引擎（不跑模型推理，不生成任何内容）

### 网络拓扑

```
┌──────────┐      ┌──────────────┐      ┌─────────────────────┐
│  前端     │ ──── │  Kalm │ ──── │ Ollama        :11434 │
│ (浏览器)  │      │  (端口 7000)  │      │ ComfyUI        :7001 │
└──────────┘      │              │      │ TTS            :8001 │
                  │ 纯中转 JSON  │      └─────────────────────┘
                  │ 代理文件流   │
                  └──────────────┘
```

前端只能访问 Kalm（7000 端口），永远不知道后端服务的实际地址。文件通过 `/file/{service}/{path}` 获取。

### 调用链路

```
# 普通任务（HTTP 模式）
POST /interface/tasks/submit
  └─ tasks.py: 创建任务 → scheduler.submit_task()
       └─ scheduler._scheduler_loop():
            1. 按需启动/切换后端服务
            2. 从 task_executors 找到对应 executor
            3. executor(task_id, **payload)
                 ├─ image.py → ComfyUI HTTP + WS → 透传 JSON
                 ├─ llm.py   → Ollama HTTP → 透传文本
                 └─ tts.py   → TTS HTTP + 轮询 → 透传 JSON
            4. executor 调用 TaskManager.update_task() 更新状态

# 进度查询
GET /interface/tasks/{id}/status
  └─ tasks.py: TaskManager.get_task() → 返回当前状态快照

# WebSocket 进度推送（WS 模式）
GET /interface/tasks/{id}/ws
  └─ ws_proxy.py → ws_manager 注册连接 → executor 实时推送 / 兜底轮询

# 流式生成（HTTP 流式模式）
POST /interface/llm/generate-stream
  └─ stream_proxy.py → scheduler → execute_stream() → Ollama NDJSON → StreamingResponse

# 文件代理
GET /file/{service}/{path}
  └─ file_proxy.py: httpx → 后端 → StreamingResponse → 前端
```

## 三、调度器设计

### 3.1 核心机制

[src/core/scheduler.py](src/core/scheduler.py) — 全局单例，单线程 FIFO 消费。

| 组件 | 实现 | 说明 |
|------|------|------|
| 队列 | `queue.Queue` | 标准库线程安全 FIFO |
| 消费线程 | `threading.Thread(daemon=True)` | 单一消费者，保证串行 |
| 超时二分法 | `get(timeout=idle_timeout)` / `get(timeout=None)` | 有服务时带超时（等待复用），无服务时无限阻塞（省 CPU） |
| 执行器注册 | `importlib.import_module` + `getattr` | 全由 `config.yaml` 驱动 |
| 优雅关闭 | Poison pill (`None`) + FastAPI lifespan 事件 | 应用退出时自动调用 |

### 3.2 服务生命周期

```
任务入队 → get(timeout) 返回 → 检查 required_service
  ├─ required == current → 复用，直接执行
  ├─ required != current → stop(current) → start(required) → 执行
  └─ required is None    → 纯逻辑任务，直接执行

执行完毕 → 回到循环顶部 → get(timeout=idle_timeout)
  ├─ timeout 内有新任务 → 复用服务
  └─ timeout 到期       → stop(current_service) → get(timeout=None) 待机
```

`service_controller.py` 使用引用计数防止同服务多任务场景下误杀进程。

### 3.3 单线程设计的合理性

本项目目标部署环境为单 RTX 4090 消费级显卡 + 64GB 内存。在此约束下：

- GPU 绑定的 ComfyUI 同一时刻只能跑一个任务，队列并发无意义
- Ollama 推理也消耗 GPU，与 ComfyUI 并发会导致显存溢出
- 序列化执行是最安全的策略，优先保证系统不宕机

如果未来需要 LLM 和 TTS 与图像生成并行（它们不竞争 GPU），可在调度器中加入任务类型白名单机制，允许特定类型跳过单线程限制。当前保持简单。

## 四、Executor 设计规范

每个 executor 必须遵循 `提交 → 等待 → 透传` 的契约。

### 4.1 函数签名

```python
def execute(task_id: str, **payload):
```

- `task_id`：由 TaskManager 创建，唯一标识
- `payload`：前端提交的原始数据，调度器解包后传入
- 无返回值 — 结果通过 `TaskManager.update_task()` 写回

### 4.2 必须做的事

1. 从 `payload` 提取需要的字段，原样或重组后 POST 给后端
2. 轮询或 WebSocket 监听后端进度
3. 调用 `TaskManager.update_task(task_id, STATUS_RUNNING, result=...)` 实时更新中间状态
4. 完成后调用 `TaskManager.update_task(task_id, STATUS_SUCCESS, result=...)` 写入最终结果

### 4.3 绝不能做的事

- 下载文件到本地磁盘
- 改写后端返回的 URL
- 处理音频/图片/视频字节流
- 判断"代理模式"还是"下载模式"
- 任何形式的业务逻辑

### 4.4 三个实现

| Executor | 后端协议 | 进度追踪 | 代理模式 |
|----------|---------|---------|---------|
| [image.py](src/core/executors/image.py) | HTTP POST /prompt + WebSocket | WS 实时推送 `executing`/`progress` | HTTP + WS |
| [llm.py](src/core/executors/llm.py) | HTTP POST (Ollama) | 无中间进度 | HTTP |
| [llm.py](src/core/executors/llm.py) `execute_stream` | HTTP NDJSON 流 (Ollama) | 逐 token 流式推送 | HTTP 流式 |
| [tts.py](src/core/executors/tts.py) | HTTP POST + 轮询 GET /status | 每 2 秒轮询，透传完整状态 JSON | HTTP |

### 4.5 添加新任务类型

1. 在 `src/core/executors/` 下新建 `xxx.py`，实现 `execute(task_id, **payload)`
2. 在 `config.yaml` 的 `tasks:` 下添加新条目：
   ```yaml
   tasks:
     my_new_task:
       service: "SomeService"          # 对应 services.yaml，如无需服务可省略
       executor:
         module: "src.core.executors.xxx"
         function: "execute"
   ```
3. 如需新后端服务，在 `services.yaml` 中添加定义
4. 重启服务即可生效，无需改动调度器代码

## 五、文件代理

### 5.1 设计

```
GET /file/{service_name}/{file_path}
```

- `service_name` — 对应 `services.yaml` 的服务名（如 `ComfyUI`、`TTS`、`Ollama`），大小写不敏感
- `file_path` — 文件在后端服务上的路径，支持多层路径

### 5.2 行为

- 从 `service_controller` 查找服务地址
- 拼接 `{host}:{port}/{file_path}` 向后端发起 GET
- 透传 `Content-Type`、`Content-Length`、`ETag` 等响应头
- 透传前端 `Range` 请求头（支持断点续传和视频拖动）
- 流式返回 `StreamingResponse`，不缓存到磁盘

### 5.3 前端使用示例

```javascript
// 后端 TTS 服务返回的 audio_url 是 "/tasks/abc123/line_1.wav"
// 前端拼接代理 URL 后请求:
fetch("/file/tts/tasks/abc123/line_1.wav")

// 后端 ComfyUI 返回的 output 包含 filename "ComfyUI_00001_.png"
fetch("/file/comfyui/view?filename=ComfyUI_00001_.png&subfolder=&type=output")
```

## 六、API 参考

### 6.1 统一响应格式

```json
{
  "status": "success" | "error",
  "message": "...",
  "data": { ... }
}
```

### 6.2 端点明细

#### GET /interface/ping
健康检查，含下游服务连通性。
```json
// 响应
{
  "status": "success",
  "data": {
    "ping": "pong",
    "service": "Kalm",
    "downstream_services": [
      {"name": "Ollama", "port": 11434, "running": true, "pid": 12345},
      {"name": "ComfyUI", "port": 7001, "running": false, "pid": null},
      {"name": "TTS", "port": 8001, "running": false, "pid": null}
    ],
    "all_healthy": false
  }
}
```

#### POST /interface/tasks/submit
统一任务提交。`task_type` 直接使用配置中的任务名：`comfyui` | `llm` | `tts`。
```json
// 请求 — ComfyUI 图像生成
{
  "task_type": "comfyui",
  "payload": {
    "workflow": { ... },
    "final_output_node_id": "9"
  }
}
// 请求 — LLM 文本生成
{
  "task_type": "llm",
  "payload": {
    "model": "gpt-oss:20b",
    "prompt": "你好"
  }
}
// 请求 — TTS 语音生成
{
  "task_type": "tts",
  "payload": {
    "path": "/v1.5/generate",
    "text": "要合成的文本",
    "speaker": "解说-魔魔"
  }
}
// 响应
{
  "status": "success",
  "data": {
    "task_id": "a1b2c3d4-...",
    "status": "queued",
    "track_mode": "ws",
    "hint": "Connect to ws://host:port/interface/tasks/a1b2c3d4-.../ws for real-time progress"
  }
}
```
`track_mode` 告诉前端如何接收结果：

| track_mode | 含义 | 前端操作 |
|-----------|------|---------|
| `ws` | WebSocket 实时进度 | 连接 `ws://host:port/interface/tasks/{task_id}/ws` |
| `poll` | HTTP 轮询 | 轮询 `GET /interface/tasks/{task_id}/status` |
| `stream` | HTTP 流式 | 响应本身就是 NDJSON 流（走 `/llm/generate-stream`） |

#### GET /interface/tasks/{task_id}/status
轮询任务状态。
```json
// 运行中
{"status": "success", "data": {"status": "running", "result": {"processed_lines": 5, "total_lines": 10}}}
// 成功
{"status": "success", "data": {"status": "success", "result": { ... 后端返回的完整数据 ... }}}
// 失败
{"status": "success", "data": {"status": "failed", "result": {"message": "..."}}}
// 不存在
{"status": "error", "message": "Task 'xxx' not found"}
```

#### GET /interface/llm/models
列出可用 LLM 模型。

#### POST /interface/llm/generate
LLM 文本生成（便捷入口，等效于 `POST /tasks/submit` 配 `task_type=llm`）。

#### GET /interface/tasks/{task_id}/ws
WebSocket 进度推送。连接后接收后端实时进度消息，任务完成时自动关闭。
```
ws://host:port/interface/tasks/{task_id}/ws
```
消息格式：
```json
// 进度更新（ComfyUI 透传）
{"type": "executing", "data": {"node": "9", "prompt_id": "..."}}
{"type": "progress", "data": {"value": 5, "max": 20}}

// 任务完成
{"type": "task_complete", "status": "success", "result": {...}}

// 任务失败
{"type": "task_failed", "message": "error reason"}
```

#### POST /interface/llm/generate-stream
LLM 流式生成。返回 `application/x-ndjson` 流，逐 token 推送，与 Ollama 原生 `/api/generate` (stream=true) 协议一致。
```json
// 请求
{
  "model": "gpt-oss:20b",
  "prompt": "写一首诗"
}
```
响应为 NDJSON 流，每行一条 JSON：
```json
{"model":"gpt-oss:20b","response":"春","done":false}
{"model":"gpt-oss:20b","response":"眠","done":false}
...
{"model":"gpt-oss:20b","response":"","done":true,"total_duration":1234567890}
```

#### GET /file/{service_name}/{file_path}
通用文件代理，参见第五章。

## 八、任务状态生命周期

```
                 POST /tasks/submit
                       │
                       ▼
                  ┌─ pending ──┐
                  │            │
                  ▼            │
              scheduler.submit_task()
                  │
                  ▼
              ┌─ queued ─┐
              │          │
              ▼          │
         调度器线程取出任务
         (服务已就绪)
              │
              ▼
         ┌─ running ─┐
         │  可多次更新  │  ← executor 通过 update_task() 实时推送进度
         │   (result  │    ─ 前端通过轮询 GET /tasks/{id}/status 获取
         │   被覆盖)  │
         └─────┬─────┘
               │
          ┌────┴────┐
          ▼         ▼
      success    failed
```

注意：
- `running` 状态下 `result` 会被**整体覆盖**（不是追加）。前端应始终取最新值。
- 服务重启后所有任务状态丢失（内存存储），任务文件由 `task_cleanup.py` 定期清理。

## 九、与 echo-drama-services 的对比

| | echo-drama-services (旧) | Kalm (新) |
|---|---|---|
| 定位 | 业务编排 + 内部加工 | 纯中转透传 |
| Executor 职责 | 下载文件、改写 URL、判代理模式 | 提交 → 轮询 → 透传 JSON |
| 文件处理 | storyboard 专属 media_proxy | 通用 `/file/{service}/{path}` |
| 音频处理 | audio_processor.py (pydub) | 不存在 |

| 配置驱动 | 部分硬编码端点后缀 | 端点也由配置驱动 |
| 包管理 | 无 `__init__.py` | 标准包结构 |

## 十、开发与运行

```powershell
# 安装依赖
pip install -r requirements.txt

# 启动（默认 7000 端口）
python main.py

# 或使用脚本
.\run.ps1

# 指定端口
.\run.ps1 -Port 8080
```

### 启动顺序

1. Kalm 本身（`python main.py`）
2. 后端服务由 Kalm 按需自动启动（首次任务到达时），按需关闭（空闲超时后）
3. 如后端服务由外部管理，设置 `manage_lifecycle: false` 并自行启动

### 常见问题

**Q: 为什么任务一直 pending/queued 不执行？**
A: 检查 `GET /interface/ping`，确认对应后端服务状态为 `running`。如果调度器日志显示 `safe_start` 超时，检查服务脚本路径和 `ready_keyword` 是否正确。

**Q: 文件代理返回 502？**
A: 后端服务未启动或端口不对。检查 `services.yaml` 中对应服务的 `host:port`。

**Q: 如何让 LLM 请求和图像生成并行？**
A: 当前串行设计基于显存安全考虑。如需并行，修改 `scheduler.py` 的 `_scheduler_loop` 增加按 task_type 的并发判断（LLM/TTS 不占 GPU，可另起线程），需注意 `TaskManager` 已线程安全。

## 十一、客户端使用指南

Kalm 提供三种代理模式，每种对应不同的后端协议。

### 11.1 模式总览

| 模式 | 端点 | 适用任务 | 数据方向 |
|------|------|---------|---------|
| HTTP 轮询 | `POST /tasks/submit` + `GET /tasks/{id}/status` | `comfyui` `llm` `tts` | 请求 → task_id → 轮询结果 |
| WebSocket 推送 | `GET /tasks/{id}/ws` | `comfyui`（实时进度） | WS 连接 → 持续推送进度 → 完成信号 |
| HTTP 流式 | `POST /llm/generate-stream` | `llm_streaming` | 请求 → NDJSON 流 → 逐 token |

### 11.2 HTTP 轮询模式

适用于 ComfyUI 图像生成、LLM 普通生成、TTS 语音生成。

```javascript
const BASE = "http://127.0.0.1:7000/interface";

// 1. 提交任务
const res = await fetch(`${BASE}/tasks/submit`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    task_type: "comfyui",
    payload: {
      workflow: { /* ComfyUI workflow JSON */ },
      final_output_node_id: "9",
    },
  }),
});
const { data } = await res.json();
const taskId = data.task_id;  // "a1b2c3d4-..."

// 2. 轮询状态直到完成
async function pollUntilDone(taskId) {
  while (true) {
    await new Promise(r => setTimeout(r, 2000)); // 2 秒间隔
    const res = await fetch(`${BASE}/tasks/${taskId}/status`);
    const { data } = await res.json();

    if (data.status === "success") return data.result;
    if (data.status === "failed") throw new Error(data.result.message);

    // data.status === "running" — 可读取中间进度
    console.log("进度:", data.result);
  }
}

const result = await pollUntilDone(taskId);
// ComfyUI 返回: { message: "...", outputs: { "9": { images: [...] } } }
// TTS 返回:    { status: "completed", lines: [...] }
// LLM 返回:    { text: "...", model: "gpt-oss:20b" }
```

### 11.3 WebSocket 模式

适用于需要实时进度的 ComfyUI 图像生成。

```javascript
const taskId = "a1b2c3d4-...";
const ws = new WebSocket(`ws://127.0.0.1:7000/interface/tasks/${taskId}/ws`);

ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);

  switch (msg.type) {
    case "executing":
      console.log("ComfyUI 正在执行节点:", msg.data.node);
      break;
    case "progress":
      console.log(`进度: ${msg.data.value}/${msg.data.max}`);
      break;
    case "task_complete":
      console.log("任务完成:", msg.result);
      ws.close();
      break;
    case "task_failed":
      console.error("任务失败:", msg.message);
      ws.close();
      break;
  }
};

ws.onerror = (err) => console.error("WS 连接错误:", err);
```

### 11.4 HTTP 流式模式

适用于 LLM 流式文本生成，逐 token 显示。

```javascript
const BASE = "http://127.0.0.1:7000/interface";

const res = await fetch(`${BASE}/llm/generate-stream`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    model: "gpt-oss:20b",
    prompt: "写一首关于春天的诗",
  }),
});

const reader = res.body.getReader();
const decoder = new TextDecoder();
let buffer = "";

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split("\n");
  buffer = lines.pop(); // 保留不完整行

  for (const line of lines) {
    if (!line.trim()) continue;
    const chunk = JSON.parse(line);
    if (chunk.response) {
      process.stdout.write(chunk.response); // 逐 token 输出
    }
    if (chunk.done) {
      console.log("\n\n完成，耗时:", chunk.total_duration / 1e9, "s");
    }
  }
}
```

### 11.6 自适应轮询策略

多任务场景下，对每个任务独立轮询会造成请求风暴。推荐客户端维护一个任务列表，根据队列位置动态调整轮询间隔：

```
队列位置 0（正在执行）→ 每 2s 检查
队列位置 1           → 每 4s 检查
队列位置 2           → 每 6s 检查
队列位置 3           → 每 8s 检查
队列位置 4+          → 每 10s 检查
任务完成 → 整体前移，间隔重新计算
```

```python
import time

def adaptive_interval(position: int) -> float:
    intervals = [2, 4, 6, 8]
    return intervals[position] if position < len(intervals) else 10


class TaskPoller:
    """自适应轮询器：一个主循环管理所有 HTTP 轮询任务"""

    def __init__(self):
        self.tasks: dict[str, dict] = {}   # task_id -> {status, result, ...}
        self._last_check: dict[str, float] = {}
        self._intervals: dict[str, float] = {}

    def submit(self, task_id: str):
        self.tasks[task_id] = {"status": "queued"}

    def remove(self, task_id: str):
        self.tasks.pop(task_id, None)
        self._intervals.pop(task_id, None)
        self._last_check.pop(task_id, None)

    def poll_loop(self, base_url: str, stop_event=None):
        """主循环，按需逐个检查"""
        import requests

        while stop_event is None or not stop_event.is_set():
            active = [tid for tid, t in self.tasks.items()
                      if t["status"] not in ("success", "failed")]

            if not active:
                time.sleep(1)
                continue

            for pos, tid in enumerate(active):
                now = time.time()
                interval = self._intervals.get(tid, 0)

                # 首次或位置变更，更新间隔
                new_iv = adaptive_interval(pos)
                if new_iv != interval:
                    self._intervals[tid] = new_iv
                    interval = new_iv

                # 没到时间，跳过
                if now - self._last_check.get(tid, 0) < interval:
                    continue

                self._last_check[tid] = now
                resp = requests.get(f"{base_url}/tasks/{tid}/status", timeout=5)
                data = resp.json()["data"]
                self.tasks[tid] = data

                if data["status"] in ("success", "failed"):
                    self._intervals.pop(tid, None)

            time.sleep(0.5)  # 主循环半秒 tick，实际请求由 interval 控制
```

```javascript
// JavaScript 版本
class TaskPoller {
  constructor() {
    this.tasks = new Map();
    this.lastCheck = new Map();
    this.intervals = new Map();
  }

  adaptiveInterval(position) {
    const intervals = [2000, 4000, 6000, 8000];
    return position < intervals.length ? intervals[position] : 10000;
  }

  submit(taskId) { this.tasks.set(taskId, { status: "queued" }); }
  remove(taskId) { this.tasks.delete(taskId); }

  async tick(baseUrl) {
    const active = [...this.tasks.entries()]
      .filter(([, t]) => t.status !== "success" && t.status !== "failed");

    for (const [pos, [taskId]] of active.entries()) {
      const now = Date.now();
      const interval = this.adaptiveInterval(pos);
      if (now - (this.lastCheck.get(taskId) || 0) < interval) continue;

      this.lastCheck.set(taskId, now);
      const resp = await fetch(`${baseUrl}/tasks/${taskId}/status`);
      const { data } = await resp.json();
      this.tasks.set(taskId, data);

      if (data.status === "success" || data.status === "failed") {
        this.intervals.delete(taskId);
      }
    }
  }
}
```

```javascript
async function generateImage(workflow, finalOutputNodeId) {
  const BASE = "http://127.0.0.1:7000/interface";

  // 提交任务
  const res = await fetch(`${BASE}/tasks/submit`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      task_type: "comfyui",
      payload: { workflow, final_output_node_id: finalOutputNodeId },
    }),
  });
  const { data } = await res.json();
  const taskId = data.task_id;

  // 两种方式二选一：
  // 方式A: WebSocket 实时进度（推荐）
  // 方式B: HTTP 轮询（简单，无需 WS）

  // 方式A: WS
  return new Promise((resolve, reject) => {
    const ws = new WebSocket(`ws://127.0.0.1:7000/interface/tasks/${taskId}/ws`);

    ws.onmessage = (event) => {
      const msg = JSON.parse(event.data);
      if (msg.type === "progress") {
        updateProgressBar(msg.data.value, msg.data.max);
      }
      if (msg.type === "task_complete") {
        ws.close();
        resolve(msg.result.outputs);
      }
      if (msg.type === "task_failed") {
        ws.close();
        reject(new Error(msg.message));
      }
    };
    ws.onerror = () => reject(new Error("WebSocket 连接失败"));
  });
}

async function pollTask(taskId) {
  while (true) {
    await new Promise(r => setTimeout(r, 2000));
    const res = await fetch(`http://127.0.0.1:7000/interface/tasks/${taskId}/status`);
    const { data } = await res.json();
    if (data.status === "success") return data.result.outputs;
    if (data.status === "failed") throw new Error(data.result.message);
  }
}
```
